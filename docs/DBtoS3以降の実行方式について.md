# DB → S3 身体図画像移行  
## 実装方式 比較まとめ（運用・構成シンプルさ重視）

---

## 🎯 前提条件

- 本番稼働 **前のみ使用**
- 一時的なデータ移行
- 画面不要
- できるだけ構成をシンプルにしたい
- 誰が実行しても同じ結果になる運用にしたい

---

# 🏗 比較対象

1. ローカル実行コンソールバッチ（.NET / Node）＋ AWS SDK  
2. PC上バッチ（単一スクリプト）  
3. PC上バッチ + AWS CLI  
4. AWS Lambda  

---

# ① ローカル実行コンソールバッチ（.NET / Node）＋ AWS SDK

## 🔹 概要

- .NET や Node でコンソールアプリを作成
- AWS SDK を使用して S3 に直接 PutObject
- .NET なら単一exe化可能（single-file publish）

### 実行例
BodyImageMigrator.exe --dry-run

---

## ✅ メリット

- 単一実行ファイルにできる（.NETの場合）
- 環境依存が少ない
- エラー制御・リトライ制御が柔軟に実装可能
- ログ出力（成功／失敗CSV）が容易
- 再実行オプションを実装できる（--limit, --overwrite 等）
- AWS CLIに依存しない
- 運用手順が明確（exeを実行するだけ）

---

## ❌ デメリット

- 初回ビルド工程が必要
- 開発コストはスクリプトよりやや高い

---

## 🎯 運用適性

★★★★☆（非常に高い）

---

# ② PC上バッチ（単一スクリプト：Node / Python 等）

## 🔹 概要

- 単一スクリプトで実装
- npm / pip 等で依存パッケージ管理

### 実行例
node migrate.js

---

## ✅ メリット

- 作成が速い
- 柔軟性が高い
- 小規模移行なら十分

---

## ❌ デメリット

- 実行環境依存（Node/Pythonバージョン）
- npm install 等が必要
- 環境差異トラブルの可能性
- 再現性がやや低い
- 運用担当がスクリプトを誤って編集するリスク

---

## 🎯 運用適性

★★★☆☆（中程度）

---

# ③ PC上バッチ + AWS CLI

## 🔹 概要

- Base64変換はローカルで実装
- アップロードは `aws s3 cp` などCLI使用

### 実行例
aws s3 cp file.png s3://bucket/path/

---

## ✅ メリット

- AWS CLIに慣れていれば扱いやすい
- SDK実装が不要
- 単純なアップロードなら簡単

---

## ❌ デメリット

- エラー制御が弱い
- 個別メタデータ制御が面倒
- 再実行制御が難しい
- 失敗一覧の取得が困難
- CLIインストール必須
- ログの整理がしづらい

---

## 🎯 運用適性

★★☆☆☆（今回用途では低め）

---

# ④ AWS Lambda

## 🔹 概要

- LambdaでDB取得 → Base64変換 → S3アップロード
- DBがオンプレならVPC接続が必要

---

## ✅ メリット

- AWS内完結
- サーバーレスでインフラ管理不要
- IAMロール管理が自然
- 将来的な定期同期には向いている

---

## ❌ デメリット

- VPC設定が複雑（特にオンプレDB接続）
- 15分タイムアウト制限
- 大量データ移行には不向き
- CloudWatchログ確認が必要
- 一時用途には過剰設計
- トラブル時の切り分けがやや難しい

---

## 🎯 運用適性

★★☆☆☆（一時移行用途では低め）

# 🏆 結論

今回の条件：

- 本番前のみの一時移行
- シンプルな構成を重視
- 運用を分かりやすくしたい
- 再実行や失敗時対応も想定

これらを総合すると、

## ✅ 最適解は  
### 「ローカル実行コンソールバッチ（.NET）＋ AWS SDK」

---

## 🔹 理由

- 単一exe化できる（環境差異が少ない）
- 実行手順が明確
- ログ管理が容易
- 再現性が高い
- エラー制御が柔軟
- 一時用途にちょうど良い規模感

---

# 🎯 推奨実装イメージ

- .NET 8 Console
- AWS SDK for .NET
- SQL接続（Dapper等）
- 並列数制御あり
- ログ出力（成功／失敗CSV）
- オプション：
  - --dry-run
  - --limit
  - --overwrite
